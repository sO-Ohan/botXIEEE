# 5. The ROS 2 Workspace

## Why ROS 2 at all?

ROS 2 is a message-passing middleware plus an enormous robotics ecosystem.
Instead of one monolithic program, the system is small **nodes** that
exchange typed messages. We get RViz, MoveIt, joystick drivers, and TF for
free, and each of our own components stays ~200 lines.

Communication patterns you'll see in this project:

| Pattern | Example here |
|---|---|
| **Topic** (stream) | `/joint_states`, `/joy`, `/joint_jog` |
| **Service** (request/reply) | `/go_home` |
| **Action** (long-running job with feedback/cancel) | `/arm_controller/follow_joint_trajectory` |

## ROS 2 inside distrobox

CachyOS/Arch isn't a ROS target platform, so ROS 2 **Humble** lives in an
Ubuntu 22.04 **distrobox** container. Your home directory is shared, so
the code lives on the host and runs in the container:

```bash
distrobox enter ubuntu           # everything ROS happens in here
source /opt/ros/humble/setup.bash
```

USB devices (`/dev/ttyUSB0`, `/dev/input/js0`) pass through automatically.

## Workspace anatomy

```
ros2_ws/
├── src/                  ← the only part you write
│   ├── botx_description/   robot model (URDF/xacro)          [ament_cmake]
│   ├── botx_moveit_config/ MoveIt configuration               [ament_cmake]
│   ├── botx_driver/        hardware driver / trajectory exec  [ament_python]
│   └── botx_teleop/        gamepad mapping                    [ament_python]
├── build/  install/  log/  ← generated by colcon; never committed
```

Build and use:

```bash
cd ~/Documents/botXIEEE/ros2_ws
colcon build --symlink-install   # --symlink-install: Python edits apply without rebuild
source install/setup.bash        # overlay this workspace onto the ROS install
```

## Who talks to whom

```
                    /joy                /joint_jog (velocities)
  joy_node ────────────────► botx_teleop ──────────────┐
                                                       ▼
  move_group ── FollowJointTrajectory action ──► botx_driver ──► /dev/ttyUSB0 → ESP32
      ▲                                                │
      │            /joint_states (positions)           │
      └───────────────◄────────────────────────────────┤
                                                       ▼
  robot_state_publisher ──► /tf ──► RViz (and MoveIt collision checking)
```

`botx_driver` is the **single owner of robot state**. Whether a motion
came from MoveIt (trajectory action) or the gamepad (jog velocities), the
driver integrates it, publishes `/joint_states`, and streams pulse targets
to the ESP32. If the serial port is absent it simply simulates — same
topics, no hardware. That's why every demo works identically with or
without the physical arm.
